### 1.1 Two facts that shape the whole plan (verified against source this phase)

- **One invocation of `enqueue-due-reminders` runs ALL FIVE producer scans in one pass** -
  `enqueueMedicationDue`, `enqueueTaskDue`, `enqueueTaskOverdue`, `enqueueAppointmentUpcoming`,
  `enqueueVisitUpcoming` execute in order against the whole circle on a single request. There is **no way
  to invoke just one producer type.** Therefore "exactly one expected notification" is a property of the
  **QA data**, not of the invocation: the QA circle must contain exactly one item that lands in any
  window, with everything else out of window. Recipients are resolved **per item** (a loop over resolved
  recipients), so one item that resolves to N recipients yields up to N notifications.
- **`item_claimed`, `item_completed`, `item_cancelled`, `claim_digest`, and `item_assigned` have NO
  producer** - neither an Edge Function nor a SQL producer/trigger exists (2F-5.2 V2.3a returned no
  producer function; V2.3b showed only `*_set_updated_at` and `*_collaborator_scope` triggers). These
  five types are **inert**: live SQL resolver capability (Section 5) plus app render surfaces (2F-5B)
  only. They **can** be checked at the resolver layer in Stage 1, but they **cannot** be generated by any
  producer invocation in Stage 2. The producer-generated universe is exactly:
  `medication_due`, `medication_missed`, `task_due`, `task_overdue`, `appointment_upcoming`,
  `visit_upcoming`.

## 2. Smoke-test objectives

Each objective maps to a stage below. The smoke test must prove:

1. **Responsible-owner targeting works** - an assigned item notifies its owner only (Stage 1 B2-style
   resolver check; Stage 2/3 producer check).
2. **`remote_member` receives no operational events** - excluded for `medication_due`, `task_due`,
   `item_claimed`, `claim_digest` (and all operational/awareness/digest types), SQL-enforced (Stage 1).
3. **An unassigned task sends nobody** - `task_due` / `task_overdue` resolve to an empty set; the
   producer no-ops (`if (recipients.length === 0) continue`) (Stage 1 B4; Stage 2 negative check).
4. **Unassigned medication / appointment / unlinked-visit manager fallback works** - resolver returns the
   active managers (Stage 1 B3 / B5a; B5b for visit is allow-list-asserted only, see Section 11).
5. **Manager escalation for missed dose targets managers only** - `check-missed-doses` tier-2 fires
   `data.tier='manager'` to `notification_item_managers`, only for an assigned med past the escalation
   threshold. (Design only here; `check-missed-doses` is deliberately NOT the first producer, Section 6.)
6. **New app notification labels / routes render the created rows** - the notification center resolves
   glyph, label, and deep link for producer-generated types (Section 7, Stage 3).
7. **No broad circle fan-out occurs** - `recipientsForItem` is per-item and owner-aware; only the
   documented unassigned fallbacks reach managers. No legacy circle-broadcast path
   (`circle_notification_recipients`) is exercised.
8. **Idempotency / dedupe prevents duplicate rows** - a repeated producer run collapses on
   `(user_id, dedupe_key)`; `enqueueForRecipient` returns null and no new row is created (Stage 4).
9. **Outbox is not processed until explicitly approved** - `process-notification-outbox` (the only push
   sender) stays blocked until Stage 5 (Section 9).

## 6. Stage 2 - producer-only smoke-test plan (design only)

**Design only. Do not execute.** Invoke only after Stage 1 resolver checks pass, and only under separate
explicit approval.

**Rules honored:**

- Choose exactly one low-risk producer path first.
- Prefer **`enqueue-due-reminders`** (a pure producer - it creates notification/inbox rows but sends no
  push; it does not even bundle `_shared/expo.ts`, so it is structurally incapable of sending a push per
  2F-6C).
- **Avoid `check-missed-doses` first** - it is absence-triggered (fires on the lack of a
  `medication_logs` row), has two conditional tiers (owner + tier-2 manager escalation gated by both
  `responsible_user_id` and the escalation threshold), is timezone-sensitive across today/yesterday, and
  carries medical-safety exposure (a false "not recorded" alert). Its timing windows are harder to stage.
- **Do not invoke `process-notification-outbox`.** No push in this stage.
- Use the cron secret (`NOTIFICATIONS_CRON_SECRET` via the `x-cron-secret` header) only in the future
  approved invocation; **do not reveal it** in any report, log, or chat. The handlers fail closed with 401
  without it.

**Which function is invoked first, and why:**

- **Function:** `enqueue-due-reminders`.
- **Why:** it is a producer (no push side effect), it is the simplest to reason about, and its recipient
  resolution is per-item and owner-aware. Because one invocation runs all five scans (Section 1.1), the
  QA data must be shaped so that **exactly one item is eligible in any window.**

**Chosen single item - one ASSIGNED, open `task_due`:**

- **Rationale:** an assigned task resolves to **exactly one owner** (the assigned doer) with **no manager
  fallback** (unassigned tasks go to nobody, not managers), so it cannot fan out. It avoids the
  appointment multi-lead multiplier and the medication dose-log complexity. It has an app render surface
  (label `taskOverdue`/`taskDue`, route `/tasks/{id}`).

**QA data shaping required (to guarantee exactly one notification):**

- One `care_tasks` row: `status = 'open'`, `assigned_to` = a single QA doer, `due_date` (+ optional
  `due_time`) positioned so `dueAt` falls inside `[now, now + taskLookaheadMinutes]`.
- **All other** QA medications / tasks / appointments / visits are out of every window (or absent), so no
  other scan produces a row. In particular: no medication schedule due in the medication window, no other
  task in the due or overdue window, no appointment inside any configured lead window, no planned visit in
  the visit window.

**Expected result of the single invocation:**

- **Expected created notification type:** `task_due`.
- **Expected recipient:** the single assigned QA doer (one recipient -> one notification). `remote_member`
  and elder excluded by SQL.
- **Expected `data` fields on the created row** (from `enqueue-due-reminders`):
  - `entity`: `'task'`
  - `itemId`: `<task.id>`
  - occurrence keys: `taskId = <task.id>`, `dueDate = task.due_date`, `dueTime = task.due_time`,
    `dueAt = <ISO>`
  - deep link: `/tasks/<task.id>`
  - dedupe key: `task:<task.id>:<task.due_date>:<due_time or 'none'>`
- **Expected outbox row state:** the producer calls `enqueue_notification` (SECURITY DEFINER), which per
  the `_shared/enqueue.ts` comment handles dedupe + outbox + quiet-hours deferral **in the DB**. Whether a
  queued outbox/delivery row exists depends on the recipient's push preference and an active token.
  Per-**device** delivery rows are only materialized when `process-notification-outbox` runs its fan-out
  phase (Stage 5). No outbox is **processed** in this stage.
- **Expected push:** **none.** `enqueue-due-reminders` cannot send push.

**Alternative single items (if a QA task is unavailable):** one assigned `appointment_upcoming` positioned
so **only one** configured lead fires (its `starts_at` minus each lead in `REMINDER_CONFIG.appointmentLeadMinutes`
must place a single lead in-window; otherwise multiple leads create multiple rows), or one assigned
`medication_due` with a single active schedule time in-window and no prior `medication_logs` row. `task_due`
remains the recommended first path for its clean single-recipient, no-fallback profile.

### 7.1 App render surfaces relevant to producer-generated rows (2F-5B)

`notificationRoute()` resolves a tap target in priority order: explicit `deep_link` column -> `data.deepLink`
-> the type's `fallbackRoute`. Relevant mappings:

- `task_due` -> route `/tasks/<id>` (producer sets explicit deep link); type-list fallback `/tasks`.
- `task_overdue` -> label key `notifications.types.taskOverdue`, glyph `clock`/warning, fallback `/tasks`;

- `visit_upcoming` -> label key `notifications.types.visitUpcoming`, glyph `visit`, fallback `/visits`;

- `medication_due` -> deep link `/medications`.
- `appointment_upcoming` -> deep link `/appointments/<id>`.
- (Inert, no producer: `item_claimed`/`item_completed`/`item_cancelled` have `fallbackRoute = null` and
  rely on an explicit producer deep link; `claim_digest` -> `/available-to-claim`.)
